# Session Management

## What Is a Session?

HTTP is stateless. Sessions bind authenticated identity to subsequent requests via a **session ID** — a cryptographically random token that is temporarily equivalent to the user's strongest credential.

Compromise of the session ID → full session hijacking without needing the password.

## Session ID Requirements

| Property              | Minimum                                                      |
| --------------------- | ------------------------------------------------------------ |
| Entropy               | ≥ 64 bits                                                    |
| Length (hex encoding) | ≥ 16 hex chars                                               |
| Generation            | CSPRNG (not Math.random(), not timestamp)                    |
| Content               | Opaque — no PII, no decodable meaning                        |
| Name                  | Generic (not `PHPSESSID`, `JSESSIONID` — fingerprints stack) |

64-bit entropy at 10,000 guesses/sec with 100,000 active sessions → ~585 years to brute-force one.

## Cookie Security Attributes

```
Set-Cookie: id=<value>; Secure; HttpOnly; SameSite=Strict; Path=/app
```

| Attribute              | Effect                                      |
| ---------------------- | ------------------------------------------- |
| `Secure`               | HTTPS only — prevents MitM interception     |
| `HttpOnly`             | No JS access — prevents XSS theft           |
| `SameSite=Strict/Lax`  | Limits cross-site sending — CSRF mitigation |
| No `Expires`/`Max-Age` | Non-persistent — gone on browser close      |
| Narrow `Domain`        | Prevent cross-subdomain sharing             |

## Session Life Cycle Security

### Strict Mode

Only accept session IDs generated by the server. Reject unknown IDs (prevent fixation).

### Regenerate After Privilege Change

Mandatory: new session ID after login, role escalation, password change. Prevents session fixation attacks.

### Session Fixation Attack

Attacker sets known session ID in victim's browser before login → after login, attacker knows valid authenticated ID. Prevented by always regenerating session ID at authentication.

## Session Expiration

| Type             | Description                          | Typical Range |
| ---------------- | ------------------------------------ | ------------- |
| Idle timeout     | Expire after N minutes of inactivity | 2-30 min      |
| Absolute timeout | Expire after N total hours           | 4-8 hrs       |
| Renewal timeout  | Auto-rotate session ID mid-session   | App-specific  |

- Enforce timeouts **server-side** — never rely on client-side timers
- On expiration: invalidate server-side session object; clear cookie client-side
- Use `Cache-Control: no-store` on responses containing session IDs
- Use `Clear-Site-Data: "cache", "cookies", "storage"` on logout

## Storage Alternatives

| Storage             | XSS Risk               | Persists?  | Recommendation             |
| ------------------- | ---------------------- | ---------- | -------------------------- |
| Cookie (`HttpOnly`) | Low (JS can't read)    | Optional   | Default choice             |
| `localStorage`      | High (JS-readable)     | Yes        | Never for session tokens   |
| `sessionStorage`    | High (JS-readable)     | Tab close  | Never for session tokens   |
| Web Workers         | None (isolated memory) | Page close | For JS-inaccessible tokens |

## Session Attack Detection

- Rate-limit requests testing different session IDs
- Alert on session ID anomalies: IP change, User-Agent change mid-session
- Log session lifecycle events (creation, renewal, destruction) — log salted hash of ID, not raw ID
- Optionally reject simultaneous logins from same account

## Session Fixation vs Hijacking

- **Session Fixation:** Attacker forces a known session ID before authentication
- **Session Hijacking (sidejacking):** Attacker steals an active session ID after authentication
- Both prevented by: HTTPS + `Secure` cookie + regenerating ID on login

## WAF Capabilities

ModSecurity + OWASP CRS can enforce `Secure`/`HttpOnly` attributes, detect fixation, and implement sticky sessions — useful for legacy apps where source cannot be changed.
